🕓 자바스크립트 Date 객체와 타임존, 그리고 뜻밖의 함정

항공편 출도착 시간을 다루다 보면, “시간”은 단순한 숫자가 아니라 “지역”과 밀접하게 연결되어 있다는 걸 실감하게 됩니다.
이번 포스트에서는 실제로 내가 겪었던 자바스크립트 Date 객체와 타임존 후킹 이슈를 바탕으로, 내부 동작 방식을 정리하고 해결 과정을 공유하려 합니다.

🚨 문제의 시작: 일부 사용자만 이상하게 보이는 출도착 시각

우리 시스템에서는 항공편 정보를 응답으로 받을 때, 각 공항의 로컬 시간을 함께 내려줍니다.
예를 들어 인천 → 뉴욕 편이라면 다음과 같은 데이터가 내려오는 식입니다.

{
  "departure": "2025-10-12T10:30:00",
  "arrival": "2025-10-12T21:15:00",
  "departureAirport": "ICN",
  "arrivalAirport": "JFK"
}


기존 레거시 코드에서는 이 값을 단순히 문자열로 쪼개서(split)

연도: 1~4번째,

월: 6~7번째,

일: 9~10번째

이런 식으로 “하드코딩”된 형태로 표시하고 있었어요.

UI 리뉴얼을 하면서 날짜 포맷팅이나 요일 표시를 좀 더 편하게 하기 위해
이 문자열을 Date 객체로 변환해서 format 함수로 처리하도록 바꿨습니다.

🧭 내 위치가 달라도 괜찮아야 했다

항공편 출도착 시각은 각 공항의 로컬 시간이므로,
내가 서울에 있든 뉴욕에 있든 Date로 변환해서 포매팅하면
“그 지역 기준의 시간”이 유지돼야 합니다.

예상대로 대부분의 환경에서는 문제없이 잘 동작했습니다.
하지만 특정 사용자에게만 시간대가 엉뚱하게 표시되는 현상이 발생했습니다.

🕵️ 원인은 ‘타임존 확장 프로그램’

문제의 사용자는 브라우저 확장 프로그램을 통해 타임존을 “UTC”로 강제로 지정해두고 있었어요.
예를 들어 “Local Timezone Changer” 같은 브라우저 확장 도구들이 있죠.

그런데 신기하게도, macOS에서 시스템 지역을 UTC로 바꾼 것과는 달리,
이 확장 프로그램을 썼을 때만 Date가 이상하게 보였습니다.

⚙️ Date 객체의 실제 동작 방식

여기서부터 본격적으로 파보기 시작했습니다.

자바스크립트의 Date는 항상 내부적으로 UTC 기준의 시간 값을 저장합니다.
즉,

문자열을 전달받을 때 → 전달된 시간과 현재 시스템의 로컬 타임존을 참고해서 UTC로 변환해 저장

출력(toString, toLocaleString 등)할 때 → 저장된 UTC 값을 현재 시스템 타임존으로 다시 변환해 보여줌

요약하자면:

Date는 항상 UTC로 저장하고, 로컬 시간대로 보여준다.

이 말은, 시스템이 UTC든 KST든 상관없이
“입력과 출력이 같은 타임존 기준이라면” 문제없이 동일한 시간이 표시되어야 한다는 뜻입니다.

🧩 그런데 확장 프로그램은 다르게 동작했다

문제는 확장 프로그램이 시스템 수준이 아닌 브라우저 레벨에서 후킹을 하기 때문이었습니다.

즉,

입력(new Date(...)) 시에는 여전히 원래 시스템 타임존을 기준으로 처리되지만,

출력(toString 등) 시점에만 후킹된 가짜 타임존(예: UTC) 으로 변환되었던 거죠.

이 때문에, 같은 코드라도 “입력은 서울 기준, 출력은 UTC 기준”처럼
타임존이 어긋나는 비정상적인 상황이 만들어졌습니다.

🧩 해결 전략: 시간 + 타임존을 함께 관리하자

그래서 내린 결론은 단순했습니다.

“시간 값 자체뿐 아니라, 그 시간의 로컬 타임존 정보까지 함께 저장하자.”

예를 들어 다음처럼 응답을 처리하는 거죠:

{
  time: "2025-10-12T10:30:00",
  timezone: "Asia/Seoul"
}


그다음 이걸 표기할 때는 date-fns-tz의 formatInTimeZone을 사용합니다.

import { formatInTimeZone } from 'date-fns-tz';

formatInTimeZone("2025-10-12T10:30:00", "Asia/Seoul", "yyyy-MM-dd HH:mm:ss");


이 함수는 지정된 타임존으로 직접 계산하기 때문에,
시스템이나 브라우저 타임존 후킹에 전혀 영향을 받지 않습니다.

🎯 결과적으로 얻은 교훈

처음엔 “브라우저 지역을 바꾸든, 확장 프로그램을 쓰든 똑같이 동작해야지?”라고 생각했는데
실제로는 후킹 방식에 따라 입력/출력 시점이 달라져 전혀 다른 결과가 나올 수 있었습니다.

결국 항공 도메인에서는 공항이 곧 타임존이라는 점을 상기하고,
“타임존을 무시하는 것이 아니라, 명시적으로 고정하는 방식”으로 해결했습니다.

✈️ 마무리하며

이 경험을 통해 배운 건,

“자바스크립트의 Date는 항상 UTC를 저장하고, 표시할 때 로컬 타임존을 적용한다.”
하지만 “브라우저가 로컬 타임존을 믿을 수 없는 상황이라면, 직접 지정하라.”

시간은 단순한 값이 아니라, 어디서의 시간인지가 항상 함께 따라다녀야 한다는 걸 깨달았습니다.
그리고 이 단순한 원칙이 실제로는 수많은 버그를 예방하는 핵심이었어요.

단순히 `Date` 객체로 시간을 처리했을 뿐인데, 예상과는 전혀 다른 결과가 나왔다.

## 문제 상황

API에서 받은 데이터 중 하나가 다음과 같았다.

```json
"departureUtc": "2025-10-12T00:30:00Z"
```

처음엔 이렇게 UTC로 들어오는 줄 알았는데, 실제 응답값에는 **Z가 없었다.**

```json
"departureUtc": "2025-10-12T00:30:00"
```

즉, 이건 **“로컬 타임 기준의 단순 문자열”**이었다.  
그래서 브라우저에서 단순히 `new Date("2025-10-12T00:30:00")`으로 생성하면  
실행 환경(사용자의 로컬 타임존)에 따라 다르게 해석되는 문제가 발생했다.

---

## 해결 방법 – `formatInTimeZone`

이 문제는 `date-fns-tz` 라이브러리의 `formatInTimeZone`을 사용해서 해결했다.

공식 문서에는 이렇게 적혀 있다.

> **formatInTimeZone** — Formats a date in the provided time zone, regardless of the system time zone.

즉, **시스템의 로컬 시간대와 무관하게 지정한 타임존으로 날짜를 포매팅해주는 함수**다.  
나는 이 함수에서 타임존 인자를 비워뒀는데,  
그 경우 기본적으로 UTC로 동작하는 것으로 보였다.  
결과적으로, 환경에 따라 다르게 보이던 시간 문제가 해결됐다.

---

## 🧭 지역(Region) 변경과 타임존

처음엔 “사용자가 브라우저 지역을 바꾸면 또 문제가 생기지 않을까?” 싶었다.  
하지만 실제로는 그렇지 않았다.

맥이나 윈도우에서 **지역(Region)을 변경하면**,  
단순히 날짜 형식(locale)만 바뀌는 게 아니라  
**시스템 타임존(Time Zone)** 자체가 함께 변경된다.  

그래서 예를 들어 같은 코드라도 시스템 지역을 바꾸면 아래처럼 다르게 보인다.

### 🇰🇷 한국 시간대일 때
```js
new Date("2025-10-12T09:30:00").toString()
// Sun Oct 12 2025 09:30:00 GMT+0900 (Korean Standard Time)
```

### 🇺🇸 뉴욕 시간대일 때
```js
new Date("2025-10-12T09:30:00").toString()
// Sun Oct 12 2025 09:30:00 GMT-0400 (Eastern Daylight Time)
```

즉, **브라우저 지역을 바꾸면 타임존 자체가 바뀌기 때문에**  
`Date` 객체도 그에 맞춰 정상적으로 동작한다.  
결국 사용자 입장에서는 시간대에 맞는 시간으로 자동 변환되므로,  
실제 서비스에서는 별다른 문제가 없었다.

## 🧩 덤 – `console.log(date)` vs `console.log(date.toString())`

디버깅하면서 헷갈렸던 부분이 하나 더 있었다.  
아래 두 코드의 출력이 달랐던 것이다.

```js
console.log(new Date("2025-10-12T09:30:00"));
console.log(new Date("2025-10-12T09:30:00").toString());
```

이유는 단순하다.  
`console.log(date)`는 실제로 **Date 객체를 그대로 로깅**하기 때문에  
브라우저 콘솔이 나중에 이 객체를 문자열로 “렌더링”할 때  
시점이나 포맷이 달라질 수 있다.

반면 `toString()`을 호출하면 **현재 시스템 타임존 기준으로 즉시 문자열 변환**을 하기 때문에  
보다 명확한 값이 출력된다.  
즉, 두 표현이 달라 보이는 건 자바스크립트의 시간 계산 문제가 아니라  
**브라우저의 출력 방식 차이**일 뿐이다.

## 🧠 확장 프로그램의 타임존 후킹 이슈

테스트 중에는 또 다른 케이스가 있었다.  
바로 **브라우저 확장 프로그램을 이용해 타임존을 임의로 변경했을 때**였다.

맥에서 지역을 바꾸는 경우엔 문제없이 일관된 결과가 나왔는데,  
확장 프로그램으로 타임존을 바꾸면 `Date` 객체가 “표기만 이상하게” 보이는 현상이 있었다.

그 이유는 **확장 프로그램의 후킹 방식이 OS 수준의 타임존 변경과 다르기 때문**이다.

- 확장 프로그램은 대개 `Date.prototype.toString`, `Intl.DateTimeFormat`, `getTimezoneOffset` 등을 **가로채서 출력만 바꾸는 방식**을 사용한다.
- 반면 `new Date()`로 객체를 생성할 때는 **시스템 타임존 기준으로 내부 UTC 타임스탬프를 계산**한다.
- 즉, 생성 시점에는 실제 OS 타임존이 쓰이고, 출력 시점에는 확장 프로그램이 후킹한 타임존이 적용된다.

그 결과,  
**저장된 UTC 값은 그대로인데, 표시할 때만 후킹된 타임존이 적용되어 “표기만 어긋나 보이는”** 상황이 생긴다.

따라서 확장 프로그램을 통한 타임존 후킹은 내부 값까지 바꾸지 않기 때문에  
시스템 수준의 지역 변경과는 다르게 작동한다는 점을 유념해야 한다.

---

## ✅ 정리

- `Z`(UTC 표기)가 없는 ISO 문자열은 **로컬 시간으로 해석**된다.  
- `formatInTimeZone`을 사용하면 시스템 타임존에 영향받지 않고 **명시적인 시간대 기준 포매팅**이 가능하다.  
- 브라우저나 OS 지역을 변경하면 타임존도 함께 바뀌므로 실제 동작에는 문제가 없다.  
- `console.log(date)` vs `console.log(date.toString())`의 차이는 **출력 포맷 시점**의 차이일 뿐이다.  
- 확장 프로그램 후킹은 출력만 후킹하기 때문에, 생성 시점의 타임존과 표시 시점의 타임존이 달라질 수 있다.

---

하지만 원리를 정확히 이해하면 생각보다 단순하다.  
핵심은 “**타임존이 언제 적용되는가**”를 명확히 구분하는 것이다. -->
